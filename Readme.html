Graphical Bar ELITE


By GreenHammerBro

Introduction

This is a guide on how to setup a display of a value of something as a graphical bar that you see in many games. Unlike some hacks like SMWCP2 and brutal Mario, which are 1-unit per tiles, this package contains code to display multiple units per tile (8 units normally), as well as representing a ratio of something with any number (1-65535) as the max.

The purpose of the code contained here is to represent something as a graphical bar to be used on various things like health, progress, percentage of something, etc. as opposed to merely displaying numbers.

The resources you need

  • Uberasm tool, needed for codes to run every frame.
  • Lunar Magic, for level testing purposes.

Recommended resources

  • Super Status Bar patch, SMW's status bar (or, heads-up display (HUD)) space is really small.

Installation (test run)

  1. First make any necessary changes to the defines in GraphicalBarDefines. Assuming you testing this out on a clean ROM with nothing else besides LM making modifications, it is likely you don't have to edit anything. If you're planning to use the Super Status Bar patch, open StatusBarSettings.asm and change !StatusBarFormat to $02 so that it displays properly.
  2. Now paste the folders:
    • GraphicalBarDefines (the whole folder) → Uberasm tool's main directory (same area as the .exe)
    • GraphicalBarELITE.asm → Uberasm tool's library.
  3. Open uberasm tool's list.txt, and paste these so that the codes get inserted: 
    ; Level list. Valid values: 000-1FF.
level:
104		DoubleBar.asm
105		Simple.asm
106		RoundAwayFromComplete.asm

  4. Insert the graphics by:
    1. Create the “ExGraphics” if you haven't done so by clicking on the blue mushroom in Lunar Magic.
    2. Paste the graphic bin files (from this package's “ExGraphics” folder) of your choice into the other where your game resides. Then, take out the appended characters after the “ExGFX<number>” but before the “.bin”
    3. Now in your level editor, click on the green poision mushroom to open a dialog about Layer 3 GFX bypass. On LG1, set that to use the graphic files (obviously the bin files are conveniently named to tell you what slot uses what file). Level 105 and 106 LG1s to be slot 80, but for 104, it needs LG1 to be slot 81 and LG2 to be slot 82 to accommodate the large number of tiles. Then hit OK and save to apply the changes.
  5. Now you are done:
    • Level 104 (Yoshi's house) is a double-bar, which can display two “fills” overlapping each other. Press right a couple of times, then up a handful of times to see.
    • Level 105 (Yoshi's island 1) is a progress meter, keep heading right to see it fills up.
    • Level 106 (Yoshi's island 2) is a coin meter, once 99 is reached, it should be full. It also avoids displaying 0% if if not 0 coins and 100% less than 99 coins, a failsafe measure to avoid misleading value displays.
    If you're not using the Super Status Bar patch, the bar will be displayed over the word “TIME” and the coin symbol. The palette will be yellow on some areas, but thats intentional as SMW's status bar is very limited.

Displaying a value of any RAM value

Perhaps if you want to display something other than the provided examples above, here is the template: 
incsrc "../GraphicalBarDefines/GraphicalBarDefines.asm"
incsrc "../GraphicalBarDefines/StatusBarSettings.asm"
;^These are needed so the defines relating to the graphical bars work.

.InputRatio
	LDA <Value>						;\Quantity (example: current HP). Use RAM here.
	STA !Scratchram_GraphicalBar_FillByteTbl		;/
	LDA <Value_high_byte>					;\High byte of above. Should your value here is 8-bit or only 1 byte long,
	STA !Scratchram_GraphicalBar_FillByteTbl+1		;/use [LDA #$00 : STA !Scratchram_GraphicalBar_FillByteTbl+1].
	LDA <Max_value>						;\Max quantity (example: max HP). Can be a fixed value (#$) or adjustable RAM in-game.
	STA !Scratchram_GraphicalBar_FillByteTbl+2		;/
	LDA <Max_value_high_byte>				;\High byte above, same format as <Value_high_byte>, so do the same
	STA !Scratchram_GraphicalBar_FillByteTbl+3		;/as that if your value is 8-bit.
.InputGraphicalBarAttributes
	LDA.b #!Default_LeftPieces				;\Left and right pieces
	STA !Scratchram_GraphicalBar_LeftEndPiece		;|
	LDA.b #!Default_RightPieces				;|
	STA !Scratchram_GraphicalBar_RightEndPiece		;/
	LDA.b #!Default_MiddlePieces				;\Number of pieces in each middle byte/8x8 tile
	STA !Scratchram_GraphicalBar_MiddlePiece		;/
	LDA.b #!Default_MiddleLength				;\length (number of middle tiles)
	STA !Scratchram_GraphicalBar_TempLength			;/
.ConvertToBar
	JSL GraphicalBarELITE_CalculateGraphicalBarPercentage	;>Get ratio/percentage
	JSL GraphicalBarELITE_DrawGraphicalBar			;>get bar values.

ASM Documentation

How this works:
  1. CalculateGraphicalBarPercentage: Calculates the ratio of something over max. For example: 50/100 would output the amount of fill half of the maximum of the bar.
  2. DrawGraphicalBar: it takes a value representing how much fill in the bar, and represent them as individual tiles bytes table with each containing the set maximum amount, then the “fraction&rd; and then finally the “empty” bytes after. It is important to note that this isn't tile numbers but rather fill amounts, to be indexed into tile numbers.
  3. ConvertFillTo8x8: converts the bar fill table into tile numbers via indexing.
  4. After this, the tile numbers are written to the status bar.

Data documentation (and what you can and can't do)

Routine info

Subroutine (JSL) Description
CalculateGraphicalBarPercentage Input:
  • !Scratchram_GraphicalBar_FillByteTbl to !Scratchram_GraphicalBar_FillByteTbl+1: The quantity of something.
  • !Scratchram_GraphicalBar_FillByteTbl+2 to !Scratchram_GraphicalBar_FillByteTbl+3: The max quantity of something.
  • !Scratchram_GraphicalBar_LeftEndPiece: The number of pieces in the left end.
  • !Scratchram_GraphicalBar_MiddlePiece: Same as above but for each middle.
  • !Scratchram_GraphicalBar_RightEndPiece: Same as above but for right end (each middle byte/tile).
  • !Scratchram_GraphicalBar_TempLength: Number of middle tiles in the bar (excluding left and right end tiles), also known as the length of the bar.
Output:
  • $00 to $01: The ratio or “percentage” amout of fill in the whole bar (rounded 1/2 up).
  • Y register: Rounded towards full or empty indicator, not considered “rounded to” if the calculation (which is (Quantity * TotalMaxPieces)/MaxQuantity) lands exactly on empty or full, Y will be #$00 if:
    • Exactly full (or more, so it treats as if the bar is full if more than enough) or empty.
    • Anywhere between full or empty
    ...and when rounded:
    • Y = #$01 if rounded towards 0 (0 < FillAmount < 0.5)
    • Y = #$02 if rounded towards full ([Max-0.5] ≤ FillAmount < Max)
    See here for more information regarding rounding away from empty and full.
Destroyed/Overwritten:
  • $02 to $0F: Whole scratch RAM was destroyed by the 32x32 bit multiplication.
DrawGraphicalBar Input:
  • $00 to $01: The (total) amount of pieces filled for the whole bar (broken into each byte).
  • !Scratchram_GraphicalBar_LeftEndPiece: (valid values are #$00 to #$FF) number of pieces the left end. Also the maximum number of filled pieces before having to fill the next tile over. If set to #$00, it won't be stored in the output table.
  • !Scratchram_GraphicalBar_MiddlePiece: Same as above, but for each middle tile.
  • !Scratchram_GraphicalBar_RightEndPiece: Same as above, but for right end tile.
  • !Scratchram_GraphicalBar_TempLength: Number of middle tiles in the bar (excluding left and right end tiles), also known as the length of the bar.
Output:
  • !Scratchram_GraphicalBar_FillByteTbl to !Scratchram_GraphicalBar_FillByteTbl+x: A table that each byte containing the fill amount ordered from maxed (any number of maxed bytes), fraction (only one such byte will exist and ranges from 0 to max-1 inclusively), then empty (any number of empty bytes) The formula for the end of this table is this formula:
    X = (LeftEndExist + MiddleLength + RightEndExist) - 1
    Where LeftEndExist and/or RightEndExist will be 0 if their pieces are set to 0, and 1 if greater than, while MiddleLength is the number of middle bytes. They are ordered from full (maxed value) bytes, then fraction, then empty. Remember that this is NOT the tile number.

    Should the total amount of fill be greater than the value needed to be full, the table would treat this as if the bar is exactly full (not exceeding); capping from writing more values in. For example: trying to input 63/62 pieces filled would have the table saying 62/62.
Destroyed/Overwritten:
  • $02 to $09: $00 to $07 are often used by math routines, $08 and $09 are the amount of fill handled by each of the 3 sections.
    • $08 to $09: the amount of fill that you have input for $00 to $01 (not capped to the value to be full if greater than).

The 1-fraction tile rule

Do note that the routine outputs with only 1 partially filled byte, due to how the maths work this out. This means that when translating it to 8x8 tiles, you cannot have such graphics (like diagonal fill edge) that can take over multiple tiles when used directly:

unless you make a code that reads the amount of fill in the fraction (after executing the routine) and check if that value is above/below a certain number (that the “edge” crosses between 2 8x8 bytes), then you can overwrite the adjacent byte/8x8 next to the fraction to display a custom edge engulfing multiple.
