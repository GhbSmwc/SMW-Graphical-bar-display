Graphical Bar ELITE


By GreenHammerBro

Introduction

This is a guide on how to setup a display of a value of something as a graphical bar that you see in many games. Unlike some hacks like SMWCP2 and brutal Mario, which are 1-unit per tiles, this package contains code to display multiple units per tile (8 units normally), as well as representing a ratio of something with any number (1-65535) as the max.

The purpose of the code contained here is to represent something as a graphical bar to be used on various things like health, progress, percentage of something, etc. as opposed to merely displaying numbers.

The resources you need

  • Uberasm tool, needed for codes to run every frame.
  • Lunar Magic, for level testing purposes.

Recommended resources

  • Super Status Bar patch, SMW's status bar (or, heads-up display (HUD)) space is really small.

Installation (test run)

  1. First make any necessary changes to the defines in GraphicalBarDefines. Assuming you testing this out on a clean ROM with nothing else besides LM making modifications, it is likely you don't have to edit anything. If you're planning to use the Super Status Bar patch, open StatusBarSettings.asm and change !StatusBarFormat to $02 so that it displays properly.
  2. Now paste the folders:
    • GraphicalBarDefines (the whole folder) → Uberasm tool's main directory (same area as the .exe)
    • GraphicalBarELITE.asm → Uberasm tool's library.
    • Inside the folder ExampleUsage, copy all the ASM files and paste them in Uberasm tool's level.
  3. Open uberasm tool's list.txt, and paste these so that the codes get inserted: 
    ; Level list. Valid values: 000-1FF.
level:
104		DoubleBar.asm
105		Simple.asm
106		RoundAwayFromComplete.asm

  4. Insert the graphics by:
    1. Create the “ExGraphics” if you haven't done so by clicking on the blue mushroom in Lunar Magic.
    2. Paste the graphic bin files (from this package's “ExGraphics” folder) of your choice into the other where your game resides. Then, take out the appended characters after the “ExGFX<number>” but before the “.bin”
    3. Now in your level editor, click on the green poision mushroom to open a dialog about Layer 3 GFX bypass. On LG1, set that to use the graphic files (obviously the bin files are conveniently named to tell you what slot uses what file). Level 105 and 106 LG1s to be slot 80, but for 104, it needs LG1 to be slot 81 and LG2 to be slot 82 to accommodate the large number of tiles. Then hit OK and save to apply the changes.
  5. Now you are done:
    • Level 104 (Yoshi's house) is a double-bar, which can display two “fills” overlapping each other. Press right a couple of times, then up a handful of times to see.
    • Level 105 (Yoshi's island 1) is a progress meter, keep heading right to see it fills up.
    • Level 106 (Yoshi's island 2) is a coin meter, once 99 is reached, it should be full. It also avoids displaying 0% if if not 0 coins and 100% less than 99 coins, a failsafe measure to avoid misleading value displays.
    If you're not using the Super Status Bar patch, the bar will be displayed over the word “TIME” and the coin symbol. The palette will be yellow on some areas, but thats intentional as SMW's status bar is very limited.

Displaying a value of any RAM value

Perhaps if you want to display something other than the provided examples above, here is the template: 
incsrc "../GraphicalBarDefines/GraphicalBarDefines.asm"
incsrc "../GraphicalBarDefines/StatusBarSettings.asm"
;^These are needed so the defines relating to the graphical bars work.

.InputRatio
	LDA <Value>						;\Quantity (example: current HP). Use RAM here.
	STA !Scratchram_GraphicalBar_FillByteTbl		;/
	LDA <Value_high_byte>					;\High byte of above. Should your value here is 8-bit or only 1 byte long,
	STA !Scratchram_GraphicalBar_FillByteTbl+1		;/use [LDA #$00 : STA !Scratchram_GraphicalBar_FillByteTbl+1].
	LDA <Max_value>						;\Max quantity (example: max HP). Can be a fixed value (#$) or adjustable RAM in-game.
	STA !Scratchram_GraphicalBar_FillByteTbl+2		;/
	LDA <Max_value_high_byte>				;\High byte above, same format as <Value_high_byte>, so do the same
	STA !Scratchram_GraphicalBar_FillByteTbl+3		;/as that if your value is 8-bit.
.InputGraphicalBarAttributes
	LDA.b #!Default_LeftPieces				;\Left and right pieces
	STA !Scratchram_GraphicalBar_LeftEndPiece		;|
	LDA.b #!Default_RightPieces				;|
	STA !Scratchram_GraphicalBar_RightEndPiece		;/
	LDA.b #!Default_MiddlePieces				;\Number of pieces in each middle byte/8x8 tile
	STA !Scratchram_GraphicalBar_MiddlePiece		;/
	LDA.b #!Default_MiddleLength				;\length (number of middle tiles)
	STA !Scratchram_GraphicalBar_TempLength			;/
.ConvertToBar
	JSL GraphicalBarELITE_CalculateGraphicalBarPercentage		;>Get percentage
	JSL GraphicalBarELITE_DrawGraphicalBar				;>get bar values.
	JSL GraphicalBarConvertToTile_ConvertBarFillAmountToTiles	;>Convert tiles.
	LDA.b #!GraphicalBarPos						;\Setup address to where to draw the bar.
	STA $00								;|
	LDA.b #!GraphicalBarPos>>8					;|
	STA $01								;|
	LDA.b #!GraphicalBarPos>>16					;|
	STA $02								;/
	if !Leftwards == 0							;\Write to status bar
		JSL GraphicalBarWriteToStatusBar_WriteBarToHUD			;|
	else									;|
		JSL GraphicalBarWriteToStatusBar_WriteBarToHUDLeftwards		;|
	endif									;/

ASM Documentation

How this works:
  1. Inside GraphicalBarELITE.asm
    • CalculateGraphicalBarPercentage: Calculates the ratio of something over max. For example: 50/100 would output the amount of fill half of the maximum of the bar.
      • RAM input Description
      !Scratchram_GraphicalBar_FillByteTbl to !Scratchram_GraphicalBar_FillByteTbl+1 (16-bit) The number or amount of something out of the maximum.
      !Scratchram_GraphicalBar_FillByteTbl+2 to !Scratchram_GraphicalBar_FillByteTbl+3 (16-bit) The maximum amount something. If this value is $0000, will assume the bar is full due to dividing by zero.
      !Scratchram_GraphicalBar_LeftEndPiece, !Scratchram_GraphicalBar_MiddlePiece, !Scratchram_GraphicalBar_RightEndPiece, and !Scratchram_GraphicalBar_TempLength (8-bit each) Graphical bar attributes. The ones with the word “Piece” refer to the number of pieces or units in their respective byte or 8x8 tile. When 0, they are not included in the table. !Scratchram_GraphicalBar_TempLength is the number of repeated middle tile bytes between the left and right end tile bytes.
      Result:
      Output: Description
      $00 to $01 (16-bit) The amount of fill as a “percentage”. Calculated by Fill_Amount = RoundHalfUp(!Scratchram_GraphicalBar_FillByteTbl * NumberOfUnitsInBar / !Scratchram_GraphicalBar_FillByteTbl)
      Y register Rounding towards empty and full indicator:
      • Y = $00 if exactly full (or more) or empty.
      • Y = $01 if rounded to empty.
      • Y = $02 if rounded to full.
        • See more information here.
      $02 to $09 Overwritten by math routines. Values should not be used here.
    • DrawGraphicalBar: it takes a value representing how much fill in the bar, and represent them as individual tiles bytes table containing a number of them that is “full”, one byte holding the “fraction” and then a number of “empty” bytes after if any of them exist. If you were to add up each byte, the total would equal to the value the number of pieces you entered. The inputs are similar to the previously mentioned routine (such as requiring the attributes), but you don't write to !Scratchram_GraphicalBar_FillByteTbl to !Scratchram_GraphicalBar_FillByteTbl+3, as it is no longer needed. Just to keep in mind, the attributes are kept after running this routine, so you don't have to rewrite them again. $00 to $01 is the number of units (or pieces) filled as the input. It is important to note that this isn't tile numbers but rather fill amounts, to be indexed into tile numbers.

      Output Description
      !Scratchram_GraphicalBar_FillByteTbl to !Scratchram_GraphicalBar_FillByteTbl+X The table containing a series of bytes ordered from full, then one containing a fraction byte, then a series of empty bytes. The ending address is X = (LeftEnd + MiddleLength + RightEnd) - 1, where LeftEnd and RightEnd is 0 should they have 0 pieces on either, and MiddleLength is basically !Scratchram_GraphicalBar_TempLength if !Scratchram_GraphicalBar_MiddlePiece is a nonzero number, and 0 otherwise.
      This is what it looks like as an example:
      • Cyan means the left end tile byte.
      • Red means right end tile byte.
      • Green means the middle tile byte(s).
      • Orange means the tile byte is full or reaches maximum.
      • Yellow means the tile byte is between 0 to max-1 inclusively (known as a fraction tile).
      • Grey means the tile byte is empty.

      ;These three are what sets the maximum fill values for the byte table.

 !Scratchram_GraphicalBar_RightEndPiece:     $03
 !Scratchram_GraphicalBar_MiddlePiece:       $08

  2. Inside GraphicalBarConvertToTile.asm
    • ConvertBarFillAmountToTiles: converts the bar fill table into tile numbers via indexing (inputs the same as above and don't need to be rewritten):
      Left end Middle Right end
      • Fill amount $00 → tile number $36
      • Fill amount $01 → tile number $37
      • Fill amount $02 → tile number $38
      • Fill amount $03 → tile number $39
      • Fill amount $00 → tile number $55
      • Fill amount $01 → tile number $56
      • Fill amount $02 → tile number $57
      • Fill amount $03 → tile number $58
      • Fill amount $04 → tile number $59
      • Fill amount $05 → tile number $65
      • Fill amount $06 → tile number $66
      • Fill amount $07 → tile number $67
      • Fill amount $08 → tile number $68
      • Fill amount $00 → tile number $50
      • Fill amount $01 → tile number $51
      • Fill amount $02 → tile number $52
      • Fill amount $03 → tile number $53

      So if !Scratchram_GraphicalBar_FillByteTbl contains these values:
      !Scratchram_GraphicalBar_FillByteTbl
      relative address:      		 +0 +1 +2 +3 +4 +5 +6 +7 +8
      Table values: $03 $08 $08 $04 $00 $00 $00 $00 $00

      Will now contain these values:
      !Scratchram_GraphicalBar_FillByteTbl
      relative address:      		 +0 +1 +2 +3 +4 +5 +6 +7 +8
      Table tile number: $39 $68 $68 $59 $55 $55 $55 $55 $50

  3. Inside GraphicalBarWriteToStatusBar.asm
    • WriteBarToHUD: Writes the tiles to the HUD. Inputs are again, similar to CalculateGraphicalBarPercentage with only the attributes being used, but a new input is required:
      Input Description
      $00 to $02 (24-bit) The starting address (left end, extends rightwards) to write the tiles to the status bar.

      Remember that this is low endian (The address $7E0F09, for example are stored in each bytes as 
      $7E0000: $09
$7E0001: $0F
$7E0002: $7E
      so be careful on that.).
    • WriteBarToHUDLeftwards: Same as above but flipped to fill leftwards. Anything at the beginning of the tile table (the “left side”) will be placed on the right side, the second tile on the second rightmost spot and so on). Currently, if you try to do this without flipping the tiles via super status bar patch or editing around address $008C81 to change their YXPCCCTT properties, you end up with the left and right ends being inverted, and when the bar fills up, it fills up from left to right within each tile starting on the right and heading left.
