How a Double Bar works

You have probably seen this type of bar: in most video games that display health. This is common in fighting games and some RPG games that when the fill drops, part or all of the fill that is loss is represented as a separate fill. This represents the damage taken to the character.

There are two ways to do this using this graphical bar code:
Rapid-flicker Using differently colored fill

Flickers between two fill values every other frame (frame 0 displays the current HP, for example, while frame 1 displays the previous before-damage HP) This gives a “transparent” appearance at 60 frames per second. The game, Kirby Super Star Ultra uses this mechanic.
Very graphic-expensive, as each tile have its own combination of display of the two fills individually as layer background tiles on the same layer cannot overlap. Not recommended for use due to this. Do note that you are limited to 4 colors on a 2BPP (2 bits per pixel) graphic (palette 0 is transparent, and palettes 1-3 are the actual colors)

How this works

These bars have two values stored in memory so that they are both represented as separate fills. It's essentially two bars merged into one. The code to do this is considerably different if you want the “rapid-flicker” or “differently colored fill”.
  • Rapid-flicker: before you call JSL to DrawGraphicalBar (if you are using this on uberasm tool, you use GraphicalBarELITE_DrawGraphicalBar due to its library system), and write to $00-$01, you first check the frame counter. SMW's frame counter is RAM address $13 (counts even when frozen) and $14 (freezable frame counter, not recommended for use). And then you can determine should you use one of the two values to store to $00-$01: 
    	LDA $13			;\Frame counter [0-255] MOD 2
	AND.b #%00000001	;/
	BEQ .Frame0
	
	.Frame1
	;Odd frame
	;REP #$20
	LDA <Value1>
	BRA .Write
	
	.Frame0
	;Even frame
	;REP #$20
	LDA <Value0>
	
	.Write
	STA $00
	;SEP #$20
	JSL GraphicalBarELITE_DrawGraphicalBar
	;...
    Remove the semicolons in front of REP #$20 and SEP #$20 should your fill amount is ever a 16-bit number going over 255 (very unlikely due to the screen is 256 pixels wide and each tile is normally up to 8 pieces per tile).

  • Differently colored fill: You actually have to call the graphical bar routines (DrawGraphicalBar and maybe CalculateGraphicalBarPercentage) twice, with the first one's table being moved to another location in memory: 
    	..GraphicalDoubleBarFirstFill
	LDA !Freeram_FirstFill				;\Amount of fill for first fill
	STA $00						;|
	STZ $01						;/
	
	;Set attributes for both bars
	LDA #!Default_LeftPieces			;\Left end pieces
	STA !Scratchram_GraphicalBar_LeftEndPiece	;/
	LDA #!Default_MiddlePieces			;\Middle pieces
	STA !Scratchram_GraphicalBar_MiddlePiece	;/
	LDA.b #!Default_MiddleLength			;\Middle length
	STA !Scratchram_GraphicalBar_TempLength		;/
	LDA #!Default_RightPieces			;\Right end pieces
	STA !Scratchram_GraphicalBar_RightEndPiece	;/
	JSL GraphicalBarELITE_DrawGraphicalBar		;>Get amount of fill.

	...TransferFirstFillBar
	;Relocate FirstFill table to another location (Located at !Scratchram_GraphicalBar_FillByteTbl + !GraphicalBar_TotalTileUsed)
	;where !GraphicalBar_TotalTileUsed is 32 ($20 in hex) by default.
	PHB
	REP #$30
	LDA.w #!GraphicalBar_TotalTileUsed-1								;>Number of bytes to transfer, -1 (because byte 0 is included)
	LDX.w #!Scratchram_GraphicalBar_FillByteTbl							;>Source address
	LDY.w #!Scratchram_GraphicalBar_FillByteTbl+!GraphicalBar_TotalTileUsed				;>Destination address
	MVN (!Scratchram_GraphicalBar_FillByteTbl>>16), (!Scratchram_GraphicalBar_FillByteTbl>>16)	;>Move them
	SEP #$30
	PLB

	..GraphicalDoubleBarSecondFill
	;Thankfully the graphical bar routines does not mess up the scratch RAM inputs, therefore
	;you only need to set them once.
	LDA !Freeram_SecondFill							;\Amount of fill for second fill
	STA $00									;|
	STZ $01									;/
	JSL GraphicalBarELITE_DrawGraphicalBar					;>Get amount of fill.
	JSL GraphicalBarConvertToTile_ConvertBarFillAmountToTilesDoubleBar	;>Translate duo fill amount to tile numbers.
	LDA.b #!Default_GraphicalBarPosition
	STA $00
	LDA.b #!Default_GraphicalBarPosition>>8
	STA $01
	LDA.b #!Default_GraphicalBarPosition>>16
	STA $02
	if !StatusBar_UsingCustomProperties != 0
		LDA.b #!Default_GraphicalBarProperties
		STA $03
		LDA.b #!Default_GraphicalBarProperties>>8
		STA $04
		LDA.b #!Default_GraphicalBarProperties>>16
		STA $05
	endif
	if !Default_LeftwardsBar == 0
		JSL GraphicalBarWriteToStatusBar_WriteBarToHUD			;>Write to status bar
	else
		JSL GraphicalBarWriteToStatusBar_WriteBarToHUDLeftwards
	endif
	RTL

    Thus, assuming you left !GraphicalBar_TotalTileUsed equal to 32, both ends set to 3 pieces, middle set to 8 pieces, and middle span 7 tile bytes long, the table of values are laid out like this: 
    !Scratchram_GraphicalBar_FillByteTbl+0:  (left end)
!Scratchram_GraphicalBar_FillByteTbl+1:  (SecondFill 1st middle tile)
!Scratchram_GraphicalBar_FillByteTbl+2:  (SecondFill 2nd middle tile)
!Scratchram_GraphicalBar_FillByteTbl+3:  (SecondFill 3rd middle tile)
!Scratchram_GraphicalBar_FillByteTbl+4:  (SecondFill 4th middle tile)
!Scratchram_GraphicalBar_FillByteTbl+5:  (SecondFill 5th middle tile)
!Scratchram_GraphicalBar_FillByteTbl+6:  (SecondFill 6th middle tile)
!Scratchram_GraphicalBar_FillByteTbl+7:  (SecondFill 7th and final middle tile)
!Scratchram_GraphicalBar_FillByteTbl+8:  (SecondFill right end)
!Scratchram_GraphicalBar_FillByteTbl+9:  (unused garbage)
!Scratchram_GraphicalBar_FillByteTbl+10: (unused garbage)
!Scratchram_GraphicalBar_FillByteTbl+11: (unused garbage)
;;;;;;;;;;;;;;;;;;;;;;;;
;11 bytes later...
;;;;;;;;;;;;;;;;;;;;;;;;
!Scratchram_GraphicalBar_FillByteTbl+30: (unused garbage)
!Scratchram_GraphicalBar_FillByteTbl+31: (unused garbage)
!Scratchram_GraphicalBar_FillByteTbl+32: (FirstFill left end)
!Scratchram_GraphicalBar_FillByteTbl+33: (FirstFill 1st middle tile)
!Scratchram_GraphicalBar_FillByteTbl+34: (FirstFill 2nd middle tile)
!Scratchram_GraphicalBar_FillByteTbl+35: (FirstFill 3rd middle tile)
!Scratchram_GraphicalBar_FillByteTbl+36: (FirstFill 4th middle tile)
!Scratchram_GraphicalBar_FillByteTbl+37: (FirstFill 5th middle tile)
!Scratchram_GraphicalBar_FillByteTbl+38: (FirstFill 6th middle tile)
!Scratchram_GraphicalBar_FillByteTbl+39: (FirstFill 7th and final middle tile)
!Scratchram_GraphicalBar_FillByteTbl+40: (FirstFill right end)
    Now that both bar tiles are stored, the routine ConvertBarFillAmountToTilesDoubleBar in GraphicalBarConvertToTile.asm checks the two equivalent tiles to determine what index to use in order to convert the pairs of fill values into tile numbers (this will write to !Scratchram_GraphicalBar_FillByteTbl+0, not the other table address).

    You can set !GraphicalBar_TotalTileUsed to any number (beware of bank borders though), just be careful not to have your longest bar in your game to exceed this number (so if you have a bar that can go higher than 9 tile bytes long, and you set !GraphicalBar_TotalTileUsed to 9 or lower), else SecondFill will overwrite FirstFill and end up with glitched tiles.
Here are some notes to keep in mind:
  • The amount of fill for SecondFill does not measure from the edge of FirstFill, rather it measures from the leftmost of the bar like FirstFill would. So if you have 1 for FirstFill and 5 for SecondFill, 4 SecondFill pieces appear just after FirstFill, not 5. Should SecondFill be less than or equal to FirstFill, it simply won't display.

  • If you want something like this without having your graphics hogged up, there is an alternative way on doing this, by using “pseudo transparency”. By using a frame counter (best if it doesn't freeze on certain events, like address $13 when $9D is set in smw), make it display 2 values depending on the frame, preferably flicker every even or odd frames.
