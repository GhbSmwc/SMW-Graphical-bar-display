Giant Masked Koopa boss summary

This document is a summary of how I implement the non-static (placed on the body of the sprite rather than a fixed position on the screen) sprite-based HP bar for the giant masked koopa boss.

Getting it to represent its HP

The value representing its damage/HP

There has to be some sort of information stored of some value tracking its HP, else technically it would have infinite HP or just die in one hit:

  • This value could start at 0 increasing on damage until it hits a certain value, and it dies-- also known as a damage counter.

  • or start at a specified value and decreases towards 0, and it dies, also known as health counter.

So how do find what sprite table or any RAM being used for that? Well we can look in the ASM file to see how it tracks it:

At the top of the ASM file, there are defines, this one in particular shows something: 
; How much hits to defeat. Do note that the hit counter, $1534, increments.
!HitPoints = $09

Thankfully, the comment gives out the information that tracks its damage, which is RAM $1534, otherwise you would need to look at how this define is being used, by CTRL+F the define for its HP: “!HitPoints” and seeing how the code interacts with it. How it is being used?, well, lets take a look (this is before I edited the sprite to show its HP): 
   JSR StompPoints             ; give Mario points
    LDA.b #!HurtSFX             ;\ sound effect
    STA.w !HurtBank             ;/
    LDA #$A0                    ;\ Set ThrowFire timer
    STA !sprite_stun_timer,x    ;/
    INC !1534,x                 ; increment sprite hit counter
    LDA !1534,x                 ;\ If it has incremented to the "max", do death animation.
    CMP.b #!HitPoints           ; |
    BEQ SpriteDead              ;/

It is an 8-bit value representing a damage counter, it increments by 1 by its only damage-- stomping on it when it is standing up. As a side note, when a sprite spawns (such as from a block, or enters from the edge of the screen or during level load), all of its values are initialized, meaning its sprite table values are all zeroes. So no need to STZ !1534,x on the inital code.

We can see here that if !1534 increments, and equals to whatever the user specifies !HitPoints to (default is $09), the boss dies (not immediately, it waits until the timer goes to zero, then it performs a “desperate attack”--the boss lunges into the player, and when it crashes into a wall, it dies).

Now how do we represent this? Well displaying HP is just merely using graphics to show to the player. Therefore, we should go to its graphics routine, the area that draws stuff on to the screen.

Graphics

Like most ASM files, they often use labels and comments making it easier for the programmer to understand how the code runs. This shouldn't be hard to find its graphics routine, we can CTRL+F “Graphics” and... 
;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;;

Graphics:
    PHA
    ASL A
    TAY
    REP #$20
    LDA X_Disp,y
    STA $04
    LDA Y_Disp,y
    STA $06
    LDA Tilemap,y
    STA $08
    LDA Properties,y
    STA $0A
    SEP #$20


This is where it sets up the graphics to draw into OAM. On this method, it stores a 16-bit address ($**XXXX, using indirect-addressing, indexed, like LDA ($XX),y) onto $04-$05 for its X position displacement, $06-$07 for Y displacement, $08-$09 for tilemap, and properties for $0A-$0B, then later, stores it state in $03 (the image state showing if the koopa is walking or inside of its shell), then perform loops to write each tile information.

Notice that the routine uses whatever was stored in A (accumulator) on the first 3 lines, which is later going to be used for indexing for indirect. So my solution is store it into $0F, since that is not used for anything else at the moment, then load that up later after calculating the graphical bar.

We want to include our code to display its HP bar, but we have to do so without interfering with its code. We could write our calculation routines before the very start of the routine (before $03-$0B becomes used for a specified purpose), and move several scratch RAM out of the way or position our code away from that if our routines uses those scratch RAM (this one is a hybrid supporting double-bar (rapid-flicker)): 
Graphics:
	;[GraphicalBar_For_HP]
	;It's best to have this graphical bar code at the very start of "Graphics" so that branches would not skip
	;over this code and cause the bar to disappear.
	STA $0F		;>You cannot push, call GetDrawInfo, then pull, because GetDrawInfo contains a code that destroy the return address,
			;which will crash the game when the sprite goes offscreen.
			; This is the graphics "image state" (what it shows), separate from "Behavior State" (what it does physically) as
			;some of the frames are walking and the other is in its shell:
			;A < $04 means in shell
			;A >= $04 means standing upright
			;Since this is used before and after calling %GetDrawInfo(), I choose to store it at $0F, an unused RAM at the moment.
	;Here is the newly added code, because this is before GetDrawInfo, we can only get the bar values here and will do the OAM
	;after calling GetDrawInfo.
	;To obtain the graphical bar data, (like the amount of fill, the attributes and all that stuff), is virtually no different
	;compared to the layer 3 version. The only difference is writing to OAM as opposed to layer 3 tiles.
		.InputRatio
			;Quantity (HP)
			;$1534 is the damage counter (increase by 1 on each hit, and dies when reaching a certain value (!HitPoints)),
			;therefore this sprite uses an "inverted health system".
			;To convert damage count to HP: HP = NumberOfHitsToKill - DamageCount
			;I could use InvertQuantity, but because the sprite always take 1 damage from stomps and no other types of damage exists,
			;and that its damage counter and !HitPoints (the fixed number of hits to defeat) are 8-bit, it is not needed, as this is faster.
				LDA #!HitPoints					;\HP = NumberOfHitsToKill - DamageCount
				SEC						;|
				SBC !1534,x					;|
				STA !Scratchram_GraphicalBar_FillByteTbl	;/Store HP into quantity.
			LDA #$00						;>Set these following data to 0 since HP is 8-bit
			STA !Scratchram_GraphicalBar_FillByteTbl+1		;>High byte quantity
			STA !Scratchram_GraphicalBar_FillByteTbl+3		;>High byte max quantity
			LDA #!HitPoints						;\Max quantity
			STA !Scratchram_GraphicalBar_FillByteTbl+2		;/
		.InputGraphicalBarAttributes
			LDA.b #!Default_GiantMaskedKoopa_GraphicalBar_LeftEndPieces		;\Left end normally have 3 pieces.
			STA !Scratchram_GraphicalBar_LeftEndPiece				;/
			LDA.b #!Default_GiantMaskedKoopa_GraphicalBar_MiddlePieces		;\Number of pieces in each middle byte/8x8 tile
			STA !Scratchram_GraphicalBar_MiddlePiece				;/
			LDA.b #!Default_GiantMaskedKoopa_GraphicalBar_RightEndPieces		;\Right end
			STA !Scratchram_GraphicalBar_RightEndPiece				;/
			LDA.b #!Default_GiantMaskedKoopa_GraphicalBar_MiddleLength		;\length (number of middle tiles)
			STA !Scratchram_GraphicalBar_TempLength					;/
		.ConvertToBar
			PHX								;>Preserve sprite slot index
			JSL !CalculateGraphicalBarPercentage				;>Get percentage
			JSL !RoundAwayEmptyFull						;>Round away from empty and full for fill amount
			;$00 = HP percentage (FirstFill)
			;$1570 = Previous HP percentage (SecondFill), or the increasing fill amount to display when healing/battle starts.
			if !GiantMaskedKoopa_DisplayPreviousHP != 0
				LDX $15E9|!addr	;>Get sprite index
				LDA !1570,x
				CMP $00
				BCC ..Increase
				BEQ ..ShowCurrent

				..Decrease
					;Decrease, causes $1570 as a transperent bar to decrease towards HP percentage.
					LDA !1558,x		;\Don't decrease SecondFill until delay runs out.
					BNE ..Flicker		;/
					DEC !1570,x
					BRA ..Flicker
				..Increase
					;In this state, when $1570 less than current HP percentage,
					;do the filling animation by having $00 contain the value of $1558 every frame and the SFX.
					INC !1570,x
					LDA $13
					AND.b #%00000001
					BNE ..ShowPrev
					LDA #$23
					STA $1DFC|!addr
					BRA ..ShowPrev
				..Flicker
					;In this state, every other frame displays previous HP
					;(because HP percentage gets subtracted on every damage,
					;while $1558 keeps its value temporary across frames),
					;giving a transperent look
						LDA $13
						AND.b #%00000001
						BEQ ..ShowCurrent
					
				..ShowPrev
					LDA !1570,x		;\When filling upwards or on every odd frames, $00 will contain the value
					STA $00			;/of $1570, showing a filling animation or previous HP percentage instead of current HP percentage.
					
					..ShowCurrent
			endif
			JSL !DrawGraphicalBar						;>get bar values.
			LDA #$01							;\Use Level-sprite tileset
			STA $00								;/
			JSL !ConvertBarFillAmountToTiles				;>Convert tiles.
			PLX								;>Restore sprite slot index
			;[...]

Now note: the calculation is before we call %GetDrawInfo(), an essential routine to get the OAM index. Most of the codes here are the same as the graphical bar code for the layer 3 status bar and overworld border, but with some key differences:

  • Because its “HP” is a damage counter, we need to reverse it by doing !HitPoints - !1543 (as in, HP = NumberOfHitsToKill - HitsTaken). We could use InvertQuantity, however, its hit counter and kill check are all 8-bit value. After reversing the damage counter, we write it to !Scratchram_GraphicalBar_FillByteTbl, and have !Scratchram_GraphicalBar_FillByteTbl+1 set to $00.

  • This is using sprite tiles, not layer 3 graphics. So don't use any routines that pertains to layer 3.
To draw the bar on OAM, we must place our OAM writing code after %GetDrawInfo() (the OAM indexing and other info are needed). We can get our whatever-was-written-to-A by simply loading $0F: 
    REP #$20
    LDA X_Disp,y
    STA $04
    LDA Y_Disp,y
    STA $06
    LDA Tilemap,y
    STA $08
    LDA Properties,y
    STA $0A
    SEP #$20

    %GetDrawInfo()

    LDA !sprite_smash_status,x
    STA $03                     ; | $03 = index to frame start (0 or 4)

    PLA
    CMP #$04
    BCS AlternateFrames
    LDA #$03
    STA $0F
    LDA $14                     ;\ Frame counter ..
    LSR #3                      ; | Add in frame animation rate; More LSRs for slower animation.
    AND #$03                    ; | 01 means we animate between 2 frames (00 and 01).
    ASL #2                      ; | ASL x2 (0-4) makes it switch between the first byte and fifth byte,
    STA $03                     ;/ i.e. first animation and second animation. The result is stored into $03.

$03-0B is being used before and after %GetDrawInfo(). Therefore they shouldn't be modified by the routine in between, therefore it's okay to have this code marked in red placed after %GetDrawInfo().

As a reminder, %GetDrawInfo() output these info:
OutputNumber of BytesDescription
Y register1 (not stored)Oam index
$001X position, relative to screen border
$011Y position, relative to screen border

After %GetDrawInfo(), we should set up all the values for the inputs for routines !CountNumberOfTiles, and !DrawSpriteGraphicalBarHoriz or !DrawSpriteGraphicalBarVert, and just restore some values for the sprite's code afterwards (such as loading $0F that we preserved).

8x8 and 16x16 tiles used together

Like I said, most sprites on SMWC is made up of 16x16, and the bar itself are 8x8s, so we need to manually set the size bit: 
    LDA ($08),y
    STA $0302|!Base2,x
    
	;[GraphicalBar_For_HP] Manually set tile size for the main body of the sprite
	;Because most sprites are composed of 16x16 OAM tiles, and the bar being 8x8s, The GMK boss is NO exception.
	;It is really not that hard, just preserve X or Y, depending on the sprite (GMK boss uses X, in this example),
	;then take X or Y, whichever holds the OAM index (increments of 4) TXA or TYA, LSR #2, TAX or TAY to convert
	;to slot numbering (increments of 1), and SET $0460, indexed by slot numbering's bit 1 (the 2nd last bit from the right) to 1.
	PHX			;>Preserve OAM index
	TXA			;\Convert OAM index numbering to slot numbering (increments of 1)
	LSR #2			;|
	TAX			;/
	LDA $0460|!addr,x	;\Force the size bit of the OAM extra bit to be 16x16 instead of potentially be 8x8
	ORA.b #%00000010	;|
	STA $0460|!addr,x	;/
	PLX			;>Restore OAM index

However, since XY are swapped due opcode limitations: LDA ($xx),x does not exist, we simply use the X register instead of Y.

Finish OAM write

Because we set the size bit, the FinishOAMWrite (either by calling JSL $01B7B3|!BankB or %FinishOAMWrite()), the Y input must be $FF, and for the number of tiles, thankfully, this isn't too hard, the number of tiles to write, minus 1, is stored in $0F already. It is easy to simply add by the number of tiles the bar uses: 
		LDY #$FF                    ; Y ends with the tile size .. 02 means it's 16x16 (Edit: the bar is 8x8s, and the body of the sprite is 16x16, so a mixture was needed; Y=$FF means Manuel)
		LDA $0F                     ; A -> number of tiles drawn - 1.
		CLC
		ADC.b #!GiantMaskedKoopa_GraphicalBar_TotalTiles
                                ; I drew 2 tiles, so 2-1 = 1. A = 01.
    JSL $01B7B3|!BankB          ; Call the routine that draws the sprite (finish OAM write).
